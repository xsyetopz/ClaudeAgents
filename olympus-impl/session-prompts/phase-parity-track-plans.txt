Goal:
Create three large implementation-plan tracks for Olympus OAL parity, instead of many tiny plans.

This is planning only. Do not implement feature code.

Product direction:
Olympus is a PiCodingAgent-first framework. The goal is to take good OAL ideas and re-author them as Olympus-owned Pi-native features: extensions, hooks, skills, prompts, commands, safety checks, acceptance fixtures, package reports, project-local artifacts, RTK-backed output compaction, and bounded Olympus agent modules.

Naming direction:
Use Greek/Olympus module names where appropriate:
- Athena: planning / architecture review
- Themis: policy / approval gates
- Apollo: verification / diagnostics
- Hephaestus: approved implementation/apply
- Hermes: handoff / routing / compact summaries
- Hestia: project-local state / continuity
- Aegis: runtime safety hooks
- Moirai: task ordering / dependency graph

Greek names are codenames for authority-bounded modules, not roleplay personas. Every module must have a sober public description and concrete safety contract.

RTK direction:
RTK is a first-class Olympus integration target for command-output compaction and token-efficiency. Olympus should use RTK when available and design RTK-backed command/output paths explicitly. In-house compactors are fallbacks for cases where RTK is unavailable, unsupported, or insufficient for a specific output shape. The model must not be allowed to silently avoid RTK when RTK is available.

Do not reintroduce:
- OAL compatibility mode
- Codex/Claude/OpenCode provider renderers
- provider plugin sync
- direct copying of OAL or third-party code
- uncontrolled swarms
- global mutation by default
- unsafe-host as a normal workflow
- future non-Pi harness planning

- broker permits approved read-only request
- quota pressure emits warning but does not invent limits

Combine these themes:
- RTK/token-efficiency and output compaction preserving decision-critical context
- better package catalogs, status handoffs, and acceptance reports
- quota and usage awareness where it affects reporting/display

Design:
- RTK is the primary compaction backend when available
- detect RTK and expose RTK status in Olympus reports
- route supported shell/read/grep/find/test/log workflows through RTK-backed command paths where possible
- provide explicit Olympus commands/prompts/hooks that encourage or enforce RTK-backed usage instead of hoping the model chooses it
- provide in-house fallback compactors only for unsupported/missing RTK cases
- record a degraded/fallback reason whenever RTK is unavailable or bypassed
- request raw/verbose fallback when RTK compression drops decision-critical context
- git diff/status/log compaction
- test output compaction
- grep/rg output compaction
- package-manager output compaction
- stack trace/error summary compaction
- raw-output reference/fallback
- compact/verbose modes
- saved-bytes/token-estimate audit
- richer package catalog/spec output
- package risk reports
- project status handoffs
- acceptance reports
- drift reports
- install/uninstall summaries
- LLM-readable handoff files
- quota profile/status display

Rules:
- do not silently skip RTK when it is available
- RTK-backed compaction is preferred over in-house fallback for supported cases
- compaction must not hide safety-critical failures
- preserve exit status, failing test names, error messages, changed files, deleted files, redaction notices, blocked-policy reasons, policy warnings, and approval decisions
- token compaction fails open only after recording fallback/degraded reason
- safety policy fails closed
- if RTK is unavailable, report degraded compaction status
- if RTK output drops decision-critical context, request raw or verbose fallback
- do not hardcode opaque provider limits as truth
- official quota values may be cited but treated as unstable
- unknown quota is labeled unknown, not fabricated
- reports must be deterministic
- reports must not include secrets
- catalog/spec reflects implemented commands only

- RTK detected when present
- RTK-backed command path is recommended or used for supported output-heavy workflows
- fallback/degraded reason is recorded when RTK is unavailable
- failing tests remain visible after compaction
- secret-looking output redacted before summary
- raw output fallback exists
- compacted diff preserves filenames and deletion markers
- status detects manifest drift
