# Product Surface

Olympi has one primary runtime, two visible modes, and one internal support
layer. The primary runtime is inside Pi: Olympi is a first-party Pi
extension/harness that wraps Pi workflows. Pi invokes it from the default
project-local extension entrypoint, explicit global registration, or explicit
`pi -e` path. The `olympi` CLI is a
bootstrap/admin entrypoint for install, uninstall, doctor, status, reports,
and scoped developer verification. Project state lives under `.pi/olympi/**`; global `~/.pi/agent/**` writes
require explicit `--global` confirmation/provenance and are never the default.

## Default user workflow

Purpose: get coding-agent work done without learning Olympi internals.

```sh
olympi install --dry-run
olympi install --apply
pi
/olympi-goal fix the failing install smoke and verify it
/olympi-plan review the changed docs
/olympi-execute run the approved verification command
/olympi-complete record verified completion
olympi status
olympi report handoff
olympi memory status
```

The default path is Pi-native and human-present. Slash commands steer a real
`lifecycle` goal-loop state through policy, hook, skill, provenance, blocker,
and verification paths; mutations require explicit confirmation/provenance.
Completion refuses to pass until verification records and audit evidence exist.
Startup help does not expose package, hook, provenance, or debug internals.
Developer/CI CLI forms are documented as internal/admin forms under full help
only; they are not the public workflow surface.

## Developer and CI/admin workflow

Purpose: inspect and control install/admin state while keeping the normal workflow in Pi.

```sh
olympi doctor --json
olympi status --json
olympi report status --json
bun run olympi:verify -- --json
bun run olympi:catalog -- --json
```

Internal package, safety, and debug diagnostics are discoverable through full help
and repository scripts. They are not the default user workflow and should not be
documented as the primary way to use Olympi.

## Debug and internal workflow

Purpose: diagnose implementation behavior and support authoring workflows.

```sh
olympi help all
bun run olympi:verify -- --json
bun run olympi:catalog -- --json
```

Debug commands are internal/developer diagnostics, not normal workflow entrypoints. Olympi routes command execution
through RTK automatically. Unsupported commands are proxied through RTK; agents
must not bypass this path.

## Hidden/internal-only capabilities

The package APIs for goal-loop state, mutation queues, first-party resource
generation, hook pipelines, and trust proofs remain library capabilities unless
a user or developer workflow needs a stable command. Do not add a command only
because a package function exists.

## Product notes

- Default CLI help stays short and focuses on install, status, health, and
  reports.
- Package intake, safety diagnostics, hook metadata, provenance, and debug
  utilities are available through explicit admin/developer surfaces.
- Normal workflow steering belongs to Pi slash commands and skills.

## Product rules

- User-mode commands stay few and goal-oriented.
- Power-user commands are grouped under explicit surfaces.
- Safety/status/reporting are available on demand, not dumped at startup.
- Autonomous operation is never the default; it must be explicit and gated.
- Names should describe the user action, not the internal module that handles it.
